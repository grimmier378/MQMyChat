How to Extend MQMyChat

This page walks you through five common modifications, step by step. Each tutorial builds one small feature from scratch so you can see the pattern, then apply it to your own ideas.

Before you begin, make sure you can build the plugin from source and load it in-game. A text editor with C++ syntax highlighting is helpful but not required.

Tutorial 1: Adding a New Default Channel

What you will learn: How the database stores channels, events, and filters. How to write SQL INSERT statements that follow the existing pattern. How channels, events, and filters relate to each other via foreign keys.

Default channels are the ones that appear when a user loads MQMyChat for the first time (or creates a new preset). They live in the SQLite database and are created by SeedDefaultChannels() in MyChatDatabase.cpp (line 617). This function runs a series of SQL INSERT statements wrapped in a transaction (all succeed or all fail together).

We will add a "Pets" channel that catches pet-related messages like "Your pet has been slain" and "begins to follow you".

Step 1: Understand the Structure

Every default channel needs three things inserted into the database:

  1. A channel row — the channel itself (name, ID, settings).
  2. One or more event rows — Blech patterns that decide which chat lines belong to this channel.
  3. One or more filter rows per event — color rules that control how matched lines are displayed.

Step 2: Pick a Unique channel_id

Look at the existing channels in SeedDefaultChannels(). The built-in ones use IDs 0, 2, 3, and so on. Pick an ID that is not already taken. For this example, we will use 20.

Step 3: Add the Channel INSERT

Open MyChatDatabase.cpp and scroll to the end of SeedDefaultChannels(), just before the ExecSQL("COMMIT") line. Add:

run(fmt::format(
    "INSERT INTO channels (preset_id, channel_id, name, enabled, echo, "
    "main_enable, enable_links, pop_out, locked, scale, main_font_size, tab_order) "
    "VALUES ({}, 20, 'Pets', 1, '/pet', 0, 0, 0, 0, 1.0, 16, 10)",
    presetId));

Here is what each value means:

  • preset_id — passed in as a parameter, ties this channel to the preset being created.
  • channel_id = 20 — our unique ID.
  • name = 'Pets' — the display name shown on the tab.
  • enabled = 1 — channel is active by default.
  • echo = '/pet' — command prefix for typed messages.
  • main_enable = 0 — do not echo pet messages to the Main tab.
  • tab_order = 10 — position in the tab bar (higher = further right).

Step 4: Add Events with Blech Patterns

Blech patterns use #*# as a wildcard that matches any sequence of characters (similar to * in a glob). Add two events for pet messages:

run(fmt::format(
    "INSERT INTO events (channel_row_id, event_index, event_string, enabled) "
    "VALUES ((SELECT id FROM channels WHERE preset_id = {} AND channel_id = 20), "
    "1, '#*#Your pet#*#', 1)",
    presetId));

run(fmt::format(
    "INSERT INTO events (channel_row_id, event_index, event_string, enabled) "
    "VALUES ((SELECT id FROM channels WHERE preset_id = {} AND channel_id = 20), "
    "2, '#*#begins to follow you#*#', 1)",
    presetId));

The channel_row_id subquery looks up the database row ID (not the channel_id) for our new channel within the current preset. This is necessary because SQLite auto-generates the row id when we insert the channel, so we cannot know it in advance — the subquery finds it at insert time. The event_index must be unique within the channel (1, 2, 3...).

Step 5: Add Filters with Colors

Each event needs at least one filter. The default filter (empty filter_string, index 0) controls the base color for all lines matching that event. Colors are RGBA floats from 0.0 to 1.0:

run(fmt::format(
    "INSERT INTO filters (event_row_id, filter_index, filter_string, "
    "color_r, color_g, color_b, color_a, enabled, hidden) "
    "VALUES ((SELECT e.id FROM events e JOIN channels c ON e.channel_row_id = c.id "
    "WHERE c.preset_id = {} AND c.channel_id = 20 AND e.event_index = 1), "
    "0, '', 0.4, 0.8, 0.2, 1.0, 1, 0)",
    presetId));

run(fmt::format(
    "INSERT INTO filters (event_row_id, filter_index, filter_string, "
    "color_r, color_g, color_b, color_a, enabled, hidden) "
    "VALUES ((SELECT e.id FROM events e JOIN channels c ON e.channel_row_id = c.id "
    "WHERE c.preset_id = {} AND c.channel_id = 20 AND e.event_index = 2), "
    "0, '', 0.2, 0.7, 0.5, 1.0, 1, 0)",
    presetId));

The color (0.4, 0.8, 0.2, 1.0) is a bright green. You can add additional filters with a non-empty filter_string to colorize specific sub-patterns differently (see the Combat channel in the existing code for examples with ^You, ^N3, etc.).

Step 6: Verify It Works

Build the plugin and load it in-game. To see your new channel, you need to trigger a fresh preset creation (existing presets are not modified). The easiest way:

  1. Open the Presets menu in the main window.
  2. Type a new name (e.g., "TestPets") and click "Save As New".
  3. Switch to the new preset. Your "Pets" tab should appear.
  4. Test it by triggering a pet-related message in-game (e.g., /pet sit and look for pet response text).
Note: This only affects new presets. Existing users keep their current channels. If you want to add a channel to an existing user's setup, you would need a database migration or have them create a new preset.

Tutorial 2: Adding a New Token

What you will learn: How the token substitution system works. How to add a new token to the if/elseif chain. Why tokens are mutually exclusive (each filter uses at most one). How to handle game state safely with null checks.

Tokens are placeholder strings used in filter patterns. When a chat line is being checked against a filter, the plugin replaces tokens with live game data before comparing. For example, M3 becomes your character's name, and N3 becomes the NPC name extracted from the line. This lets you write a single filter like ^M3 that means "lines starting with MY name" without hardcoding any character name.

We will add a ZN token that substitutes the current zone name.

Step 1: Open SubstituteTokens()

In MQMyChat.cpp, find SubstituteTokens() (line 830). This function takes a pattern string and the original chat line, and returns the pattern with all tokens replaced.

Step 2: Understand the if/elseif Chain

The function works as a chain of independent checks. Each block looks for a specific token in the pattern string. The first match replaces the token and returns immediately — tokens are mutually exclusive, meaning a single filter pattern should only contain one token. This is because the function returns as soon as it processes the first token it finds, so a second token in the same pattern would never be substituted.

if (replace("M3", pLocalPlayer->Name)) return result;

if (result.find("PT1") != std::string::npos)
{
    if (replace("PT1", GetPetName())) return result;
}

if (result.find("N3") != std::string::npos)
{
    // ... more complex logic to extract NPC name ...
}

// ... more tokens ...

Step 3: Add Your Token Block

Add a new block at the end of the chain, just before the final return result;:

if (result.find("ZN") != std::string::npos)
{
    std::string zoneName = "Unknown";
    if (pZoneInfo)
        zoneName = pZoneInfo->LongName;
    replace("ZN", zoneName);
    return result;
}

Step 4: Test Your Token

Build and load the plugin. Open the channel editor for any channel and add a new filter with the pattern ZN. Now, when a chat line is checked against this filter, ZN will be replaced with the actual zone name (e.g., "East Commonlands"). If the line contains that zone name, the filter matches and the line gets that filter's color. You can verify this by going to a zone and triggering a chat message that mentions the zone name.

Important: If your token needs to look up a spawn by name and the name contains spaces, convert spaces to underscores before calling GetSpawnByPartialName(). EQ stores spawn names internally with underscores (e.g., Selephra_Giztral), while DisplayedName uses spaces. See the N3 and PT3 token implementations for examples of this conversion.

Existing Tokens Reference

For context, here are the tokens already defined in SubstituteTokens(). See Chat Routing & Tokens for full details.

Token Substitution
M3Your character name
PT1Your pet's name
PT3Group member's pet name (or your pet)
M1Main Assist name
TK1Main Tank name
P3PC name extracted from line
N3NPC name extracted from line
RLRaid leader name
G1–G5Group member 1-5 name
H1Group healer name (Clr/Dru/Shm)
GP1Any group member (tries each)

In addition to tokens, filter patterns support two prefix modifiers:

Prefix Effect
NO2Invert the match. Place before a token or word to exclude lines that contain it. For example, NO2PT1 matches lines that do NOT contain your pet's name. Handled by MatchFilter() in MQMyChat.cpp.
^Anchor to the start of the line. For example, ^You matches only lines that begin with "You". Handled by PatternFind() in MQMyChat.cpp.

Filter patterns also support Lua string patterns (e.g., %d+%.?$ to match numbers at the end of a line). The plugin auto-detects Lua patterns by looking for the % escape character.

Tutorial 3: Adding a New /mychat Subcommand

What you will learn: How the command dispatcher works (sequential ci_equals checks). How to use GetArg and GetIntFromString to parse arguments. How to validate input and print feedback with WriteChatf.

We will add /mychat font <size> to change the main window font size from the command line.

Step 1: Open MyChatCommand()

In MQMyChat.cpp, find MyChatCommand() (line 101). All subcommands are handled by sequential if (ci_equals(...)) blocks.

Step 2: Add Your Block

Add this new block after the existing preset block and before the final "Unknown command" message:

if (ci_equals(arg, "font"))
{
    char sizeStr[MAX_STRING] = {};
    GetArg(sizeStr, szLine, 2);
    if (sizeStr[0] == '\0')
    {
        WriteChatf("\at[MQMyChat]\ax Usage: /mychat font <size>");
        WriteChatf("\at[MQMyChat]\ax Current font size: %d",
            g_chatEngine->settings.mainFontSize);
        return;
    }

    int newSize = GetIntFromString(sizeStr, 0);
    if (newSize < 8 || newSize > 48)
    {
        WriteChatf("\at[MQMyChat]\ax Font size must be between 8 and 48.");
        return;
    }

    g_chatEngine->settings.mainFontSize = newSize;
    WriteChatf("\at[MQMyChat]\ax Main font size set to %d.", newSize);
    return;
}

Step 3: Update the Usage Message

Find the "Unknown command" line at the bottom of MyChatCommand() and add font to the list of valid subcommands:

WriteChatf("\at[MQMyChat]\ax Usage: /mychat "
    "[show|hide|send|clear|config|reload|lock|unlock|preset|font]");

Step 4: Test It

Build and load the plugin. Try these commands in the EQ chat window:

/mychat font
/mychat font 20
/mychat font 5
/mychat font 100

The first should print the current font size. The second should change it to 20. The third and fourth should print an error because 5 and 100 are outside the valid range (8-48).

How It Works

  • GetArg(sizeStr, szLine, 2) grabs the second argument (the font size number). GetArg uses 1-based indexing, so argument 1 is "font" and argument 2 is the size.
  • ci_equals(arg, "font") does case-insensitive matching, so /mychat Font 20 and /mychat FONT 20 both work.
  • GetIntFromString safely converts a string to an integer, returning the second argument (0) if the string is not a valid number.
  • The early return after each block is important — without it, execution would fall through to the "Unknown command" handler.

Tutorial 4: Adding a New TLO Member

What you will learn: The three-part pattern for adding TLO members: (1) add an enum value in the header, (2) register it with ScopedTypeMember in the constructor, and (3) handle it in the GetMember switch statement. This is the same pattern used for every existing member.

We will add a Locked member to the MyChat TLO that returns whether the main window is locked. After this, ${MyChat.Locked} will work in macros and Lua.

Step 1: Add to the Members Enum

Open MyChatTLO.h and find the Members enum inside MyChatType. The enum gives each member a unique numeric ID that MacroQuest uses internally to dispatch property lookups. Add your new value at the end:

enum class Members
{
    Version,
    Channel,
    ChannelCount,
    Locked,        // <-- add this line
};

Step 2: Register the Member

Open MyChatTLO.cpp and find the MyChatType constructor. Add a ScopedTypeMember call for the new member:

MyChatType::MyChatType() : MQ2Type("MyChat")
{
    ScopedTypeMember(Members, Version);
    ScopedTypeMember(Members, Channel);
    ScopedTypeMember(Members, ChannelCount);
    ScopedTypeMember(Members, Locked);     // <-- add this line
    ScopedTypeMethod(Methods, Send);
}

ScopedTypeMember is a MacroQuest macro that registers the member name (derived from the enum value) so that FindMember("Locked") will return the correct ID. The string name is automatically taken from the enum value name, so the enum value Locked registers the member name "Locked". This is why the enum value name must match the desired property name exactly.

Step 3: Handle the Member in GetMember

In the same file, find the switch (static_cast<Members>(...)) block inside MyChatType::GetMember(). This switch statement is what MacroQuest calls when someone writes ${MyChat.Locked} — it looks up which enum value corresponds to "Locked" and jumps to the matching case. Add a new case:

case Members::Locked:
    if (!g_chatEngine)
        return false;

    Dest.Set(g_chatEngine->settings.windowLocked);
    Dest.Type = mq::datatypes::pBoolType;
    return true;

Step 4: Test It

Build, load the plugin, and try:

/echo ${MyChat.Locked}

It should print TRUE or FALSE. From Lua:

local locked = mq.TLO.MyChat.Locked()
print("Window locked:", locked)
Tip: For string members, use strcpy_s(DataTypeTemp, ...) and set Dest.Ptr = &DataTypeTemp[0] with Dest.Type = pStringType. For int members, use Dest.Int with Dest.Type = pIntType. Look at the existing cases in MyChatChannelType::GetMember() for examples of every return type.

Tutorial 5: Adding a New GUI Element

What you will learn: How ImGui's immediate-mode rendering works (widgets are drawn every frame, not created once). How to find the right render method. How to add interactive widgets like buttons and checkboxes. How to optionally persist new settings.

We will add a "Clear All" button to the config GUI that clears every channel's console in one click.

Step 1: Find the Right Render Method

Open MyChatRenderer.cpp. The config window is drawn by RenderConfigGUI() (line 303). Other render methods you might work in:

  • RenderMainWindow() (line 67) — the main tabbed chat window.
  • RenderPopOutWindows() (line 264) — detached channel windows.
  • RenderEditChannelGUI() (line 414) — the channel editor dialog.

Step 2: Understand the ImGui Flow

RenderConfigGUI() starts with:

void MyChatRenderer::RenderConfigGUI(MyChatEngine& engine)
{
    if (!engine.showConfigGUI)
        return;

    auto oldStyle = ImGuiTheme::ApplyTheme(engine.settings.themeIdx);

    std::string winName = fmt::format("My Chat - Config##MQMyChat_{}", engine.charName);
    ImGui::SetNextWindowSize(ImVec2(500, 400), ImGuiCond_FirstUseEver);
    if (ImGui::Begin(winName.c_str(), &engine.showConfigGUI, ImGuiWindowFlags_None))
    {
        // ... channel table is drawn here ...
    }
    ImGui::End();
}

The pattern is: Begin() opens the window, content goes inside the if block, and End() closes it. Widgets you add inside this block will appear in the config window. Unlike traditional GUIs where you create a button once, ImGui is immediate mode — your code runs every frame (~60 times per second), and each call to ImGui::Button() both draws the button and returns true if it was clicked that frame.

Step 3: Add the Button

A good place for a "Clear All" button is just before the channel table. Add it right after the ImGui::Begin() check:

if (ImGui::Begin(winName.c_str(), &engine.showConfigGUI, ImGuiWindowFlags_None))
{
    if (ImGui::Button("Clear All Channels"))
    {
        if (engine.mainConsole)
            engine.mainConsole->Clear();

        for (auto& [id, ch] : engine.settings.channels)
        {
            if (ch.console)
                ch.console->Clear();
        }
    }
    ImGui::Separator();

    // ... existing channel table code continues here ...

Step 4: Common ImGui Widgets

Here are the most useful widgets you might want to add elsewhere:

Widget Code What It Does
Checkbox ImGui::Checkbox("Label", &boolVar) Toggles a boolean value
Button if (ImGui::Button("Click Me")) { ... } Returns true when clicked
Text Input ImGui::InputText("Label", buf, sizeof(buf)) Editable text field
Integer Input ImGui::InputInt("Label", &intVar) Number field with +/- buttons
Separator ImGui::Separator() Horizontal divider line
Same Line ImGui::SameLine() Puts the next widget on the same row
Small Button ImGui::SmallButton("x") Compact button, useful for table cells

Step 5: Wire to Engine Settings

If your new widget should persist its value across sessions, store it in the engine settings struct and make sure the database layer saves and loads it. For one-shot actions like "Clear All", no persistence is needed.

Step 6: Verify It Works

Build and load the plugin. Open the config window (/mychat config). You should see a "Clear All Channels" button at the top. Send some test messages to a few channels, then click the button — all channel consoles and the main console should be empty.

Common Pitfalls

Before you start modifying MQMyChat, be aware of these gotchas that can cause subtle bugs or crashes:

Always Use AppendToConsole

Never call console->AppendText() directly when writing chat output. Always use engine.AppendToConsole(console, text, color) instead. This method is the single chokepoint for all console output in the plugin. It handles timestamp injection (stripping existing [HH:MM:SS] timestamps and optionally prepending a colored timestamp via multiple AppendText calls with explicit MQColor values per segment). This multi-segment approach is critical: an earlier implementation used a single string with MQ color codes (\ay[\aw...\ay]\ax), but the \ax reset code overrode the defaultColor parameter passed to AppendText, causing all message text to render white instead of the filter-assigned color. The current approach gives each piece (brackets, time digits, message text) an explicit color and preserves the original message color. If you bypass AppendToConsole(), your output will not have timestamps even when the user has them enabled, and the behavior will be inconsistent with every other output path. Every call site in the plugin — RouteToChannel, SendToChannel, spam channel output, and local echo — goes through AppendToConsole().

WriteChatf Reentrancy

Do not call WriteChatf() inside Blech event processing. Here is why: the plugin hooks both OnWriteChatColor and OnIncomingChat. Both hooks feed every chat line into the Blech pattern engine. If your Blech callback calls WriteChatf(), that new line triggers OnWriteChatColor again, which re-enters Blech, causing infinite recursion or corrupted state. For example, if you added WriteChatf("Matched a line!") inside BlechCallback, that message would itself be fed back into Blech, match again, print again, and so on until the stack overflows and the game crashes. If you need to log debug output, write to a file instead or use DebugSpewAlways().

Spawn Names: Underscores vs. Spaces

EverQuest uses two name formats internally:

  • pSpawn->Name uses underscores: Selephra_Giztral
  • pSpawn->DisplayedName uses spaces: Selephra Giztral

GetSpawnByPartialName() expects underscores. If you are searching for a name extracted from a chat line (which uses spaces), convert spaces to underscores first:

std::string searchName = nameFromChat;
for (char& c : searchName)
    if (c == ' ') c = '_';
PlayerClient* pSpawn = GetSpawnByPartialName(searchName.c_str());

EMU vs. Live Chat Format Differences

Chat message formats differ between EMU (emulated) and Live servers. For example:

  • EMU: Grimmier says 'Hello' (no comma after "says")
  • Live: Grimmier says, 'Hello' (comma after "says")

If you are writing Blech patterns for say/tell/shout events, you may need to account for both formats. The wildcard #*# in #*#says#*# handles both cases naturally.

Color Format: Database vs. Memory

Colors are stored differently in the database and in memory:

  • Database (SQLite): RGBA floats from 0.0 to 1.0 (e.g., 0.972549, 0.541176, 0.996078, 1.0)
  • Memory (ImGui): Also 0.0-1.0 floats via ImVec4
  • MQ Console: Colors as 0-255 uint8_t values

Be careful when converting between these. If you are passing colors between MQ's console API and ImGui, you need to divide or multiply by 255.

Blech Event Registration Order

Always call UnregisterBlechEvents() before calling RegisterBlechEvents() when changing patterns. If you register new events without unregistering the old ones first, you will get duplicate matches and potentially doubled output in channels.

Preset Switching Order

When switching presets, the operations must happen in exactly this order:

  1. SaveCharacterSettings() — persist current state
  2. SetActivePreset() — update the DB to point at the new preset
  3. UnloadCharacterSettings() — tear down current channels, events, Blech patterns
  4. LoadCharacterSettings() — load the new preset's data

Skipping Save means losing unsaved changes. Skipping Unload before Load means stale Blech patterns remain registered, causing ghost matches. See the preset subcommand in Commands & TLO for the working implementation.
