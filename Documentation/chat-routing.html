Chat Routing & Tokens

This page explains how a line of chat travels from EverQuest (or another MQ plugin) into the correct MQMyChat channel tab, and how tokens and filters transform the matching along the way. If you want to understand why a line appears (or does not appear) in a particular channel, read this page start to finish.

Key terms used on this page:
  • Blech — MacroQuest's built-in text pattern matcher. It is not regex and not Lua patterns. It uses a simple wildcard syntax with # delimiters (explained in detail below).
  • ConsoleWidget — An MQ ImGui widget that displays scrollable, colored text. Each channel tab is backed by one ConsoleWidget, and there is also a "main" console that aggregates lines from all channels.
  • Event — A Blech pattern string attached to a channel. When the pattern matches a chat line, that line is a candidate for the channel.
  • Filter — An optional second-pass check on lines that already matched an event. Filters can change the display color, suppress the line entirely, or act as a whitelist.
  • Token — A short placeholder like M3 or GP1 inside a filter pattern. At match time, the plugin replaces it with a live value from the game (your character name, a group member name, etc.).
  • pLocalPlayer / pLocalPC — MQ pointers to the currently logged-in character. nullptr when no character is in the game world.
  • pTarget — MQ pointer to the currently selected (targeted) entity. nullptr when nothing is targeted.

How Chat Flows

Every line of text follows the same five-step pipeline:

EverQuest generates chat MQ plugin calls WriteChatf() | | v v OnIncomingChat() OnWriteChatColor() | | v v ProcessIncomingChat() ProcessWriteChat() | | +-------- both set m_currentLine ----+ | and m_currentColor | v v m_blech->Feed(line, len) | [Blech pattern matching] | BlechCallback() fires for each matching event | v RouteToChannel() | +--- apply filters ---+ | | discard AppendToConsole() (strip dupe timestamps, optionally prepend new colored timestamp, then call AppendText on the channel's ConsoleWidget)

Step 1: EQ or MQ produces a line

MacroQuest intercepts all chat through two plugin callbacks. OnIncomingChat handles lines generated by EverQuest itself (tells, says, group, raid, combat, etc.). OnWriteChatColor handles lines written by MacroQuest or other plugins.

Step 2: Feed into Blech

Both ProcessIncomingChat and ProcessWriteChat store the raw line pointer and color, then call m_blech->Feed(): 

void MyChatEngine::ProcessIncomingChat(const char* line, int color)
{
    if (charName.empty() || !m_blech)
        return;

    m_currentLine = line;
    m_currentColor = color;
    m_blech->Feed(line, strlen(line));
    m_currentLine = nullptr;

    // ... Spam channel handling (see below)
}

The temporary m_currentLine / m_currentColor members exist because Blech's callback signature does not carry the original line or color. The callback reads them from the engine instance.

Step 3: Blech matches patterns

Blech compares the line against every registered event pattern. If a pattern matches, Blech fires BlechCallback.

Step 4: BlechCallback routes

void CALLBACK MyChatEngine::BlechCallback(
    unsigned int id, void* pData, PBLECHVALUE pValues)
{
    if (!g_chatEngine)
        return;

    auto it = g_chatEngine->m_blechEventMap.find(id);
    if (it == g_chatEngine->m_blechEventMap.end())
        return;

    const BlechEventData& evtData = it->second;

    if (g_chatEngine->m_currentLine)
        g_chatEngine->RouteToChannel(
            evtData.channelId, evtData.eventIndex,
            g_chatEngine->m_currentLine,
            g_chatEngine->m_currentColor);
}

The Blech event ID maps back to a BlechEventData struct containing the channelId and eventIndex. These tell RouteToChannel exactly where to send the line.

Step 5: RouteToChannel applies filters and appends

This is where color, filtering, and the claimed-lines map come into play. Covered in detail in the Filter System section below.

Blech Pattern Matching (First Pass: Events)

The chat routing system has two passes. The first pass uses Blech to decide which channel(s) a line belongs to. The second pass (the filter system) decides whether to actually display the line and what color to use. This section covers the first pass.

Blech is MacroQuest's built-in string matching engine. It uses a wildcard syntax based on the # character (configured in Initialize()):

Syntax Meaning Example
#*# Match any number of any characters (like .* in regex) #*#tells you,#*# matches any line containing "tells you,"
#1#, #2#, etc. Numbered capture wildcard. Matches any text (like #*#) but captures the matched segment as a numbered argument for use in event callbacks. #1# is the first capture, #2# the second, and so on. #1# says, '#2#' captures the speaker name in #1# and the message text in #2#
Literal text Must appear verbatim You have been slain only matches that exact text
Tip: Blech patterns are not regular expressions or Lua patterns. They are a simpler, faster matching system designed for chat text. The #*# and #1# wildcards match any amount of text (zero or more characters) — they are not single-character matchers. Think of #*# as equivalent to .* in regex. For more complex matching, use filter patterns which support Lua %-style patterns.
Reference: For full Blech pattern syntax documentation, see the MacroQuest Events and Binds documentation.

Each channel has one or more events. Each event has an eventString that is the Blech pattern. For example, a "Tells" channel might have two events:

  • #*#tells you, '#*#' — matches standard tells
  • #*#told you, '#*#' — matches "You told" echoes

When either pattern matches a line, that line is routed to the Tells channel. See the Blech Event Registration section on the lifecycle page for how events are registered.

Filter System (Second Pass: RouteToChannel)

After Blech matches a line to a channel (first pass), RouteToChannel runs the second pass: it decides whether to actually display the line, and what color to use. Each event can have multiple filters that refine the decision.

How filters work

void MyChatEngine::RouteToChannel(
    int channelId, int eventIndex, const char* line, int color)
{
    auto chIt = settings.channels.find(channelId);
    if (chIt == settings.channels.end())
        return;

    ChatChannel& channel = chIt->second;

    bool filtered = false;
    MQColor outputColor(240, 240, 240, 255);
    bool hasNonDefaultFilters = false;
    bool matched = false;

    for (const auto& evt : channel.events)
    {
        if (evt.eventIndex != eventIndex)
            continue;

        for (const auto& filter : evt.filters)
        {
            if (!filter.enabled)
                continue;

            if (filter.filterIndex == 0)
            {
                outputColor = filter.color;
                continue;
            }

            hasNonDefaultFilters = true;

            if (MatchFilter(line, filter))
            {
                if (filter.hidden)
                {
                    filtered = true;
                    break;
                }

                outputColor = filter.color;
                matched = true;
                break;
            }
        }
        break;
    }

    if (filtered)
        return;

    if (hasNonDefaultFilters && !matched)
        return;

    // ... Consider channel special case, then append
}

Filter evaluation rules

Filter Index Role Behavior
0 Default (fallback) filter Sets the default output color. Has an empty filterString, so it always "matches." Skipped during the actual match loop.
1..N Match filters Each has a pattern string and a color. The first matching filter wins.

The decision logic:

  1. Start with the default color from filter[0].
  2. If there are non-default filters (index 1+), iterate them.
  3. First match wins. If a matching filter has hidden = true, the line is discarded entirely.
  4. If non-default filters exist but none matched, the line is also discarded. This means adding filters acts as a whitelist: only lines matching at least one filter pass through.
  5. If no non-default filters exist at all, the line passes through with the default color.
Heads up: When you add your first filter to an event, you are effectively saying "only show lines that match this filter." All non-matching lines will disappear from that channel. This is intentional but can surprise beginners.

Appending to consoles

After a line passes filter evaluation, it is appended to the channel's own ConsoleWidget via AppendToConsole() — the central output method that handles timestamp injection. If the channel has mainEnable = true (the default), the line is also appended to the main "All" console through the same method. This is how lines from multiple channels can all appear together in the main tab. 

if (channel.console)
    AppendToConsole(channel.console, line, outputColor);

if (channel.mainEnable && mainConsole)
    AppendToConsole(mainConsole, line, outputColor);

Consider channel special case

Before appending, if the channel is named "Consider" and a target is currently selected, the output color is overridden based on the target's con color (how the mob "considers" to you, indicating its difficulty relative to your level): 

if (ci_equals(channel.name, "consider") && pTarget)
{
    int conVal = ConColor(pTarget);
    switch (conVal)
    {
    case CONCOLOR_GREY:      outputColor = MQColor(153, 153, 153); break;
    case CONCOLOR_GREEN:     outputColor = MQColor(0, 255, 0);     break;
    case CONCOLOR_LIGHTBLUE: outputColor = MQColor(94, 180, 255);  break;
    case CONCOLOR_BLUE:      outputColor = MQColor(0, 0, 255);     break;
    case CONCOLOR_WHITE:     outputColor = MQColor(255, 255, 255); break;
    case CONCOLOR_YELLOW:    outputColor = MQColor(255, 255, 0);   break;
    case CONCOLOR_RED:       outputColor = MQColor(230, 26, 26);   break;
    default: break;
    }
}

Claiming the line

After a line is appended to a channel console, it is recorded in the m_claimedLines map with a 0.5-second TTL. This is how the Spam channel knows which lines have already been handled. 

std::string lineStr(line);
m_claimedLines[lineStr] = ClaimedLine{
    std::chrono::steady_clock::now()
        + std::chrono::milliseconds(
            static_cast<int>(CLAIMED_TTL * 1000))
};

Token Substitution (SubstituteTokens)

Tokens let you write filter patterns that adapt to the current game state. Instead of hard-coding your character's name into a filter, you write M3 and the plugin substitutes it at match time.

SubstituteTokens takes a pattern string and the current chat line, performs one token replacement, and returns the result. Only the first token found is replaced (the method returns early after each substitution). This means you cannot combine multiple different tokens in a single filter pattern. For example, M3 hits N3 would only resolve M3 and leave N3 as literal text. If you need to match on multiple dynamic values, use separate filters.

Tokens are checked in this order: M3, PT1, PT3, M1, TK1, P3, N3, RL, G1 through G5, H1, GP1. The first token found in the pattern string is the one that gets substituted.

Token reference

Token What it becomes Example result Source
M3 Your character name "Grimmier" pLocalPlayer->Name
M1 Main assist name "Tankguy" Iterates group members, checks IsMainAssist()
PT1 Your pet's name "Xaanit" GetSpawnByID(pLocalPlayer->PetID)
PT3 Group member's pet name "Jobin" (a pet whose master is in your group) Extracts name from line, converts spaces to underscores, checks MasterID against group members
GP1 Any group member "Healbot" Tries each group member's name in the pattern against the line; returns the one that matches
TK1 Main tank "Tankguy" Iterates group, checks IsMainTank()
RL Raid leader "Raidlead" pRaid->RaidLeaderName
G1 – G5 Group slot 1 through 5 "Clericname" pLocalPC->pGroupInfo->GetGroupMember(i)
H1 Group healer "Healbot" Finds group member with class CLR (2), DRU (6), or SHM (10); also tests against the line to pick the right one if multiple healers
N3 NPC name "a moss snake" ExtractNameFromLine + IsNPCName, with GetSpawnByPartialName fallback
P3 PC (player) name "Otherplayer" ExtractNameFromLine where the result is not an NPC and not your pet
Fallback values: If a token cannot be resolved (no group, no pet, no raid, etc.), it substitutes a descriptive placeholder like "NO PET", "NO MA", "NO TANK", "NO GROUP", "NO RAID", "NO HEALER", or "None".

Prefix modifiers

Filter patterns support two special prefixes. They are processed at different stages:

Prefix Where it is handled Effect Example
NO2 MatchFilter — stripped before token substitution and pattern matching Invert the match result. If the pattern would match, it does not; if it would not match, it does. NO2M3 matches lines that do not contain your character name.
^ PatternFind — checked after token substitution, during pattern matching Anchor the pattern to the start of the line. Only matches if the line begins with the text after ^. ^You matches "You say, 'hello'" but not "Someone tells you, 'hello'".

You can combine them: NO2^You means "match lines that do NOT start with 'You'." The processing order is: strip NO2, substitute tokens, then PatternFind sees the ^ and does an anchored match. The NO2 flag inverts the final result.

The MatchFilter method that ties it all together:

bool MyChatEngine::MatchFilter(
    const std::string& line, const ChatFilter& filter) const
{
    if (filter.filterString.empty())
        return true;

    std::string matchStr = filter.filterString;
    bool invert = false;

    if (matchStr.size() > 3 && matchStr.substr(0, 3) == "NO2")
    {
        invert = true;
        matchStr = matchStr.substr(3);
    }

    matchStr = SubstituteTokens(matchStr, line);
    bool found = PatternFind(line, matchStr);

    return invert ? !found : found;
}

Pattern Matching (PatternFind)

While Blech handles the first pass (event matching), PatternFind is the workhorse behind the second pass. After MatchFilter strips the NO2 prefix and substitutes tokens, it calls PatternFind to check whether the resulting pattern string appears in the chat line. This function supports three modes, tried in this order:

Mode 1: Lua patterns

If the pattern contains a % character, it is treated as a Lua pattern and executed via the embedded LuaJIT state: 

if (IsLuaPattern(pattern) && m_luaState)
{
    lua_getglobal(m_luaState, "string");
    lua_getfield(m_luaState, -1, "find");
    lua_pushstring(m_luaState, text.c_str());
    lua_pushstring(m_luaState, pattern.c_str());
    int status = lua_pcall(m_luaState, 2, 1, 0);
    bool found = (status == 0 && !lua_isnil(m_luaState, -1));
    lua_settop(m_luaState, 0);
    return found;
}

This gives you access to Lua's pattern language: %d for digits, %a for letters, %] for literal brackets, character classes like [%a%d], and so on.

Detection rule: IsLuaPattern checks whether a % character appears anywhere in the pattern except as the very last character. A trailing % by itself does not trigger Lua mode. However, if your filter accidentally contains a % elsewhere (for example in a URL like %20), it will be treated as a Lua pattern.

Mode 2: Anchored match

If the pattern starts with ^, it checks whether the line starts with the remaining text: 

if (!pattern.empty() && pattern[0] == '^')
{
    std::string anchor = pattern.substr(1);
    if (anchor.empty())
        return true;
    return text.size() >= anchor.size()
        && text.compare(0, anchor.size(), anchor) == 0;
}

Mode 3: Plain substring

The default. A simple std::string::find() — returns true if the pattern appears anywhere in the line: 

return text.find(pattern) != std::string::npos;

Summary

Pattern Mode Matches
%d+ damage Lua pattern One or more digits followed by " damage"
^You Anchored Lines starting with "You"
tells you, Plain substring Any line containing "tells you,"

NPC Detection

Several tokens (N3, P3, PT3) need to distinguish between NPC names and player names in a chat line. Two helper functions handle this.

ExtractNameFromLine

Scans the line for known keywords that indicate who is speaking or acting, and extracts the name that appears before the keyword: 

static std::pair<bool, std::string>
    ExtractNameFromLine(const std::string& line)
{
    static const char* keywords[] = {
        "pet tells you",
        "tells you,",
        "says to you,",
        " says, '",
        " says '",       // EMU format
        " says,",
        " whispers,",
        " shouts,",
        " slashes",
        " pierces",
        " kicks",
        " crushes",
        " bashes",
        " hits",
        " tries",
        " backstabs",
        " bites",
        " begins",
    };
    // ...finds keyword in line, extracts name before it
}

The function returns a pair: a boolean indicating whether the name is an NPC, and the extracted name string. It calls IsNPCName to make the NPC determination.

EMU support: The keyword list includes both " says, '" (live format) and " says '" (EMU format without comma) so the plugin works on emulated servers.

IsNPCName

Takes a name, converts spaces to underscores (EQ stores NPC names with underscores internally), and looks it up via GetSpawnByPartialName. If the spawn exists and its Type is SPAWN_NPC, it returns true. 

static bool IsNPCName(const std::string& name)
{
    if (GetGameState() != GAMESTATE_INGAME || !pLocalPlayer)
        return false;

    std::string searchName = name;
    for (char& c : searchName)
        if (c == ' ') c = '_';

    if (SPAWNINFO* pNPC = GetSpawnByPartialName(searchName.c_str()))
    {
        if (pNPC->Type == SPAWN_NPC)
            return true;
    }

    return false;
}

N3 token fallback

The N3 token has an extra fallback. If ExtractNameFromLine does not flag the name as an NPC (maybe the name was not in the spawn list at that moment), the token handler does its own GetSpawnByPartialName lookup with the underscore-converted name. If that finds an NPC, it uses the spawn's DisplayedName as the substitution value: 

if (!isNPC && !npcName.empty())
{
    std::string searchName = npcName;
    for (char& c : searchName)
        if (c == ' ') c = '_';

    if (SPAWNINFO* pNPC = GetSpawnByPartialName(searchName.c_str()))
    {
        if (pNPC->Type == SPAWN_NPC)
        {
            isNPC = true;
            npcName = pNPC->DisplayedName;
        }
    }
}

Spam Channel

The Spam channel works as a reverse filter: it shows every line that was not claimed by any other channel. Think of it as a catch-all for anything you have not yet categorized.

How it works

  1. When RouteToChannel sends a line to any channel, it records the line text in m_claimedLines with a TTL of 0.5 seconds (CLAIMED_TTL).
  2. After m_blech->Feed() returns in ProcessIncomingChat, the method checks whether the line was claimed.
  3. If the line is not in m_claimedLines, it gets appended to the Spam channel console.
  4. CleanExpiredClaims() runs every 500 ms in OnPulse to remove old entries and keep memory usage bounded.
// In ProcessIncomingChat, after m_blech->Feed():
if (enableSpam)
{
    std::string lineStr(line);
    if (m_claimedLines.find(lineStr) == m_claimedLines.end())
    {
        auto spamIt = settings.channels.find(CHANNEL_SPAM);
        if (spamIt != settings.channels.end()
            && spamIt->second.console)
        {
            AppendToConsole(spamIt->second.console,
                line, MQColor(240, 240, 240));
            if (spamIt->second.mainEnable && mainConsole)
                AppendToConsole(mainConsole,
                    line, MQColor(240, 240, 240));
        }
    }
}

Note that even the spam channel routes through AppendToConsole(), so spam lines also receive timestamps when the timestamps option is enabled.

Constants

From MyChatChannel.h:

static constexpr int   CHANNEL_SPAM           = 9000;
static constexpr int   CHANNEL_RESERVED_START  = 9000;
static constexpr float CLAIMED_TTL             = 0.5f;
  • CHANNEL_SPAM (9000) is the reserved channel ID for the Spam channel.
  • CHANNEL_RESERVED_START (9000) prevents user-created channels from getting IDs in the reserved range.
  • CLAIMED_TTL (0.5 seconds) controls how long a line stays "claimed." This is deliberately short — it only needs to survive until ProcessIncomingChat finishes checking.
Note: The Spam channel only processes lines from ProcessIncomingChat (EQ chat), not from ProcessWriteChat (MQ plugin output). This avoids flooding the Spam tab with plugin debug messages.

AppendToConsole and Timestamps

Every piece of text that appears in any MQMyChat console — whether it came from Blech-routed chat, the spam channel, SendToChannel(), or local echo — passes through a single method: AppendToConsole(). This method is the chokepoint where timestamp injection happens. No code in the plugin calls console->AppendText() directly anymore; everything goes through this method.

Method signature

void MyChatEngine::AppendToConsole(
    const std::shared_ptr<mq::imgui::ConsoleWidget>& console,
    std::string_view text, MQColor color,
    bool appendNewLine = true);

How it works

When settings.timeStamps is false, the method simply forwards the text and color directly to console->AppendText() with no modifications.

When settings.timeStamps is true, the method does three things before writing:

  1. Strip existing timestamps: Some incoming chat lines from EverQuest already contain timestamps in the format [HH:MM:SS]. To avoid doubled timestamps (like [14:30:22] [14:30:22] Healbot tells you...), the method scans the text for a bracket-delimited timestamp pattern. It looks for a [ character, then checks if positions +3 and +6 are colons and position +9 is ]. If this pattern is found, the entire [HH:MM:SS] segment (plus any trailing space) is removed from the text.
  2. Get the current time: Calls the C standard library time() and localtime_s() to get the current local time, then formats it as HH:MM:SS using strftime().
  3. Prepend a colored timestamp using multiple AppendText calls: Instead of building a single string with MQ color codes, the method makes four separate AppendText calls, each with an explicit MQColor value:
    • "[" in yellow MQColor(255, 255, 0)
    • timeBuf (the HH:MM:SS digits) in white MQColor(255, 255, 255)
    • "] " in yellow MQColor(255, 255, 0)
    • The actual chat text in the original color parameter passed to AppendToConsole()
    The first three calls use appendNewLine = false so they stay on the same line; only the final call (the message text) uses the real appendNewLine value. This multi-segment approach was chosen because the previous single-string method (using MQ color codes like \ay, \aw, \ax) caused the \ax reset code to override the defaultColor parameter passed to AppendText, forcing all message text to render white instead of the filter-assigned color.

Full implementation

void MyChatEngine::AppendToConsole(
    const std::shared_ptr<mq::imgui::ConsoleWidget>& console,
    std::string_view text, MQColor color, bool appendNewLine)
{
    if (!console)
        return;

    if (settings.timeStamps)
    {
        std::string cleaned(text);
        auto bracket = cleaned.find('[');
        if (bracket != std::string::npos
            && bracket + 9 < cleaned.size()
            && cleaned[bracket + 3] == ':'
            && cleaned[bracket + 6] == ':'
            && cleaned[bracket + 9] == ']')
        {
            size_t end = bracket + 10;
            if (end < cleaned.size() && cleaned[end] == ' ')
                end++;
            cleaned.erase(bracket, end - bracket);
        }

        time_t now = time(nullptr);
        struct tm local;
        localtime_s(&local, &now);
        char timeBuf[16];
        strftime(timeBuf, sizeof(timeBuf), "%H:%M:%S", &local);

        MQColor yellow(255, 255, 0);
        MQColor white(255, 255, 255);
        console->AppendText("[", yellow, false);
        console->AppendText(timeBuf, white, false);
        console->AppendText("] ", yellow, false);
        console->AppendText(cleaned, color, appendNewLine);
    }
    else
    {
        console->AppendText(text, color, appendNewLine);
    }
}

Where AppendToConsole is called

Every output path in the plugin routes through this method. Here is the complete list:

Call Site File What It Outputs
RouteToChannel() MQMyChat.cpp Blech-matched chat lines to channel consoles and the main console
ProcessIncomingChat() (spam) MQMyChat.cpp Unclaimed lines to the spam channel and optionally the main console
SendToChannel() MQMyChat.cpp Externally injected messages (from TLO Send, /mychat send, or other plugins) to channel consoles and the main console
DrawConsole() MyChatRenderer.cpp Local echo of user-typed text (> text) to the channel console
DrawMainConsole() MyChatRenderer.cpp Local echo of user-typed text to the main console

The Timestamps toggle

The settings.timeStamps boolean is controlled by the Options > Timestamps checkbox in the main window's menu bar. It defaults to true. Changing it takes effect immediately — the next line appended to any console will either have or not have a timestamp. Previously displayed lines are not retroactively modified. The setting is persisted in the global_settings table under the key timeStamps.

Why strip existing timestamps? EverQuest can be configured to include timestamps in its own chat output. If the user has both EQ timestamps and MQMyChat timestamps enabled, the same line would show two timestamps. The stripping logic detects the [HH:MM:SS] format and removes it before prepending the MQMyChat timestamp, ensuring each line has at most one.

Putting It All Together: A Worked Example

Imagine a player named Grimmier has a "Tells" channel with this setup:

  • Event pattern: #*#tells you, '#*#'
  • Filter[0]: default color = light blue
  • Filter[1]: pattern = M3, color = green (matches lines containing your own name)
  • Filter[2]: pattern = NO2GP1, color = red, hidden = false (matches tells from people NOT in your group)

Now a tell arrives: Healbot tells you, 'incoming heal on Grimmier'

  1. OnIncomingChat receives the line.
  2. ProcessIncomingChat stores it and calls m_blech->Feed().
  3. Blech matches #*#tells you, '#*#' and fires the callback.
  4. BlechCallback looks up channelId and eventIndex, calls RouteToChannel.
  5. Filter[0] sets default color to light blue.
  6. Filter[1]: M3 becomes "Grimmier" via SubstituteTokens. PatternFind checks if "Grimmier" appears in the line. It does ("incoming heal on Grimmier"), so this filter matches. Output color becomes green.
  7. AppendToConsole() is called for the Tells channel console (and the main console if mainEnable is on). If timestamps are enabled, the method strips any existing [HH:MM:SS] timestamp, then makes four separate AppendText() calls: "[" in yellow, "14:30:22" in white, "] " in yellow, and the message text in green (the filter-assigned color). Each segment gets an explicit MQColor value, preserving the filter color on the actual message.
  8. The line is added to m_claimedLines, so the Spam channel will skip it.
