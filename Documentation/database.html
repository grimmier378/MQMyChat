Database & Settings

Last Updated: 2026-04-24

MQMyChat stores all of its configuration — channels, events, filters, presets, and per-character settings — in a single SQLite database file. This page walks you through how the database is set up, what the tables look like, and how the plugin reads and writes your settings.

The code for all of this lives in two files: MyChatDatabase.h (the class declaration) and MyChatDatabase.cpp (the implementation).

SQLite Setup

MQMyChat uses SQLite3 as its database engine. If you have never worked with SQLite before, think of it as a self-contained database that lives entirely in a single file on your hard drive. There is no separate server to install or run — the plugin links the SQLite library directly and reads/writes the file itself. That makes it simple to deploy (just one .db file) and easy to back up (copy the file).

Database File Location

The database file is stored at:

gPathResources/MyChat.db

In practice, gPathResources is your MacroQuest resources directory. This is the same folder where MQ stores other shared data files.

Connection Settings

When the MyChatDatabase class opens the database (via its Open() method), it applies three important settings called pragmas. A pragma is a special SQLite command that configures how the database engine behaves — think of them as runtime options you set before doing any real work.

PRAGMA journal_mode = WAL;
PRAGMA foreign_keys = ON;
PRAGMA busy_timeout = 3000;
Pragma What It Does
journal_mode = WAL Write-Ahead Logging. By default, SQLite uses a "rollback journal" that locks the entire database file during writes. WAL mode changes this so that new data is written to a separate log file first, then merged back into the main database later. The practical benefit: readers and writers can operate at the same time without blocking each other. This matters because MQMyChat may be reading settings on one thread while saving them on another. You may notice a MyChat.db-wal file next to the database — that is the WAL log, and it is normal.
foreign_keys = ON Enables foreign key enforcement. A foreign key is a rule that says "this column must reference a valid row in another table." For example, the channels table has a foreign key pointing to presets, which means you cannot create a channel that belongs to a preset that does not exist. SQLite has foreign key support built in, but it is turned off by default for backward compatibility — you have to explicitly enable it every time you open a connection. The schema also uses ON DELETE CASCADE on its foreign keys, which means deleting a preset automatically deletes all its channels, events, and filters.
busy_timeout = 3000 If the database is locked by another operation, wait up to 3 seconds before giving up. Without this, a busy database would return an error immediately.

Opening and Closing

Open() connects to the database file and applies the pragmas shown above. Close() does a bit more than just disconnecting — it first runs a WAL checkpoint:

PRAGMA wal_checkpoint;

This flushes any pending writes from the WAL file back into the main database file. It ensures that the MyChat.db file is fully up to date when the plugin shuts down.

Schema

The database has 8 tables. They are all created by the InitSchema() method in MyChatDatabase.cpp. Each table uses CREATE TABLE IF NOT EXISTS, so re-running the schema creation is safe — it will not destroy existing data.

Here is how the tables relate to each other. The arrows show foreign key relationships (the arrow points from the child table to the parent it references):

presets (1) ──< channels (many) [ON DELETE CASCADE] │ ├──< events (many) [ON DELETE CASCADE] │ │ │ └──< filters (many) [ON DELETE CASCADE] │ char_active_preset ──> presets [ON DELETE CASCADE] char_channel_overrides ──> presets [ON DELETE CASCADE] global_settings (standalone, no foreign keys) schema_version (standalone, tracks migration version)

The ON DELETE CASCADE annotations mean that deleting a parent row automatically deletes all its children. For example, deleting a preset removes all of its channels, which in turn removes all their events and filters. This keeps the database clean without requiring manual cleanup.

global_settings

Stores per-character key-value settings. Think of it as a simple dictionary where each character on each server has their own set of preferences.

CREATE TABLE global_settings (
    char_name TEXT NOT NULL,
    server    TEXT NOT NULL DEFAULT '',
    key       TEXT NOT NULL,
    value     TEXT NOT NULL,
    PRIMARY KEY (char_name, server, key)
);
About PRIMARY KEY: The PRIMARY KEY (char_name, server, key) is a composite primary key, meaning the combination of all three columns must be unique. This allows the same key name (like "locked") to exist once per character per server, but not twice for the same character on the same server.

The following keys are stored in this table:

Key What It Controls
lockedWhether the main window is locked (no move/resize)
timeStampsShow timestamps on chat lines
ScaleWindow scale factor
themeIdxIndex of the active theme
doLinksEnable clickable item links
mainEchoEcho command for the main chat tab
MainFontSizeFont size for the main console
LogCommandsWhether to log commands to the chat window
keyFocusWhether keyboard focus key is enabled
keyNameName of the key used for focus (e.g., "RightShift")
localEchoShow your own typed text in the channel
autoScrollAuto-scroll to newest messages

presets

A preset is a named collection of channel configurations. You might have one preset for raiding (showing raid chat, tells, and loot) and another for grouping (showing group chat and experience). The presets table stores metadata about each preset.

CREATE TABLE presets (
    id          INTEGER PRIMARY KEY AUTOINCREMENT,
    preset_name TEXT NOT NULL,
    server      TEXT NOT NULL DEFAULT '',
    description TEXT DEFAULT '',
    created_by  TEXT NOT NULL,
    created_at  TEXT NOT NULL DEFAULT (datetime('now')),
    updated_at  TEXT NOT NULL DEFAULT (datetime('now')),
    UNIQUE(preset_name, server)
);
About AUTOINCREMENT: The INTEGER PRIMARY KEY AUTOINCREMENT means SQLite automatically assigns a unique, ever-increasing ID to each new row. You never need to specify the id when inserting — the database picks the next available number. The AUTOINCREMENT keyword guarantees IDs are never reused, even after rows are deleted.

Presets are scoped to a server. This means you can have a preset called "Raiding" on one server and a completely different "Raiding" preset on another. The UNIQUE(preset_name, server) constraint enforces this — you cannot have two presets with the same name on the same server.

channels

Each preset contains one or more channels. A channel is a tab in your chat window that collects a specific type of chat (e.g., "Tells", "Guild", "Loot").

CREATE TABLE channels (
    id             INTEGER PRIMARY KEY AUTOINCREMENT,
    preset_id      INTEGER NOT NULL,
    channel_id     INTEGER NOT NULL,
    name           TEXT NOT NULL DEFAULT 'New',
    enabled        INTEGER NOT NULL DEFAULT 0,
    echo           TEXT NOT NULL DEFAULT '/say',
    main_enable    INTEGER NOT NULL DEFAULT 1,
    enable_links   INTEGER NOT NULL DEFAULT 0,
    pop_out        INTEGER NOT NULL DEFAULT 0,
    locked         INTEGER NOT NULL DEFAULT 0,
    scale          REAL NOT NULL DEFAULT 1.0,
    main_font_size INTEGER NOT NULL DEFAULT 16,
    tab_order      INTEGER NOT NULL DEFAULT 0,
    UNIQUE(preset_id, channel_id),
    FOREIGN KEY (preset_id) REFERENCES presets(id) ON DELETE CASCADE
);

Some important columns:

  • channel_id — A numeric identifier for the channel. The special value 9000 is reserved for the Spam channel.
  • echo — The command prefix used when you type into this channel's input box (e.g., /gu for guild chat).
  • main_enable — Whether this channel's messages also appear in the Main tab.
  • pop_out — Whether this channel is displayed in its own separate window instead of as a tab.
  • tab_order — The position of this channel's tab in the tab bar. Updated when you drag tabs to reorder them.

events

Each channel has one or more events. An event is a Blech pattern that determines which chat lines are routed to this channel.

CREATE TABLE events (
    id              INTEGER PRIMARY KEY AUTOINCREMENT,
    channel_row_id  INTEGER NOT NULL,
    event_index     INTEGER NOT NULL,
    event_string    TEXT NOT NULL DEFAULT 'new',
    enabled         INTEGER NOT NULL DEFAULT 1,
    UNIQUE(channel_row_id, event_index),
    FOREIGN KEY (channel_row_id) REFERENCES channels(id) ON DELETE CASCADE
);

For example, the "Tells" channel might have events with patterns like #*# tells you, '#*#' and #*# told you, '#*#'.

filters

Each event can have one or more filters. A filter is a simple string match with an associated color. When a chat line matches an event's Blech pattern and contains a filter's text, it is displayed in that filter's color.

CREATE TABLE filters (
    id             INTEGER PRIMARY KEY AUTOINCREMENT,
    event_row_id   INTEGER NOT NULL,
    filter_index   INTEGER NOT NULL,
    filter_string  TEXT NOT NULL DEFAULT '',
    color_r        REAL NOT NULL DEFAULT 1.0,
    color_g        REAL NOT NULL DEFAULT 1.0,
    color_b        REAL NOT NULL DEFAULT 1.0,
    color_a        REAL NOT NULL DEFAULT 1.0,
    enabled        INTEGER NOT NULL DEFAULT 1,
    hidden         INTEGER NOT NULL DEFAULT 0,
    UNIQUE(event_row_id, filter_index),
    FOREIGN KEY (event_row_id) REFERENCES events(id) ON DELETE CASCADE
);
Color format: Colors are stored as floating-point values between 0.0 and 1.0 in the database. When loaded into memory, the plugin converts them to uint8_t values (0–255) for use with ImGui. For example, a database value of 1.0 becomes 255, and 0.5 becomes 127.

The hidden flag lets you keep a filter defined but suppress its output. The line still matches, but it is not displayed in the channel.

char_active_preset

Tracks which preset each character is currently using.

CREATE TABLE char_active_preset (
    char_name TEXT NOT NULL,
    server    TEXT NOT NULL DEFAULT '',
    preset_id INTEGER NOT NULL,
    PRIMARY KEY (char_name, server),
    FOREIGN KEY (preset_id) REFERENCES presets(id) ON DELETE CASCADE
);

When a character logs in, the plugin looks up their active preset here. If no row exists, the plugin creates a "Default" preset and assigns it.

char_channel_overrides

Stores per-character display preferences for each channel within a preset. When multiple characters share the same preset, each character's tab order, enabled state, pop-out settings, and other display preferences are stored here independently. The preset itself remains the shared template for channel structure (names, events, filters).

CREATE TABLE char_channel_overrides (
    id          INTEGER PRIMARY KEY AUTOINCREMENT,
    char_name   TEXT NOT NULL,
    server      TEXT NOT NULL DEFAULT '',
    preset_id   INTEGER NOT NULL,
    channel_id  INTEGER NOT NULL,
    enabled     INTEGER NOT NULL DEFAULT 1,
    echo        TEXT NOT NULL DEFAULT '/say',
    main_enable INTEGER NOT NULL DEFAULT 1,
    enable_links INTEGER NOT NULL DEFAULT 0,
    pop_out     INTEGER NOT NULL DEFAULT 0,
    locked      INTEGER NOT NULL DEFAULT 0,
    scale       REAL NOT NULL DEFAULT 1.0,
    main_font_size INTEGER NOT NULL DEFAULT 16,
    tab_order   INTEGER NOT NULL DEFAULT 0,
    UNIQUE(char_name, server, preset_id, channel_id),
    FOREIGN KEY (preset_id) REFERENCES presets(id) ON DELETE CASCADE
);

On first load with a preset, no override rows exist — channels keep the preset's default display values. As the character customizes settings (reorders tabs, disables channels, changes pop-out), those changes are saved to this table. The next session loads the preset structure first, then overlays the character's display preferences from here.

What goes where: Display preferences (enabled, echo, tab_order, pop_out, locked, scale, font_size, main_enable, enable_links) go in char_channel_overrides. Structural data (channel name, events, filters) stays in the preset's channels, events, and filters tables. If a character wants different events or filters, they create a new preset.

schema_version

Tracks which database migrations have been applied. This allows the plugin to evolve the schema safely without dropping and recreating tables on every load.

CREATE TABLE schema_version (
    version INTEGER NOT NULL DEFAULT 0
);

Initialized with a single row (version = 0). The RunMigrations() method checks the current version and applies any pending migrations in order. Each migration increments the version after completing successfully.

Indexes

Four indexes are created for query performance:

CREATE INDEX IF NOT EXISTS idx_channels_preset ON channels(preset_id);
CREATE INDEX IF NOT EXISTS idx_events_channel  ON events(channel_row_id);
CREATE INDEX IF NOT EXISTS idx_filters_event   ON filters(event_row_id);
CREATE INDEX IF NOT EXISTS idx_char_channel_overrides_lookup
    ON char_channel_overrides(char_name, server, preset_id);

These speed up the nested queries used when loading settings (preset → channels → events → filters).

Preset System

Presets let you maintain multiple complete chat configurations and switch between them. Here are the key operations:

Getting the Active Preset

GetOrCreatePreset() is the entry point. It checks the char_active_preset table for the current character. If no preset is assigned, it creates one named "Default" and assigns it. This ensures every character always has a valid preset.

Listing Presets

GetPresetList() returns all presets for the current server. This is used to populate the Presets menu in the menu bar.

Creating and Copying Presets

  • SaveAsNewPreset() — Creates a brand new preset using your current channel/event/filter configuration. You provide a name, and the current settings are saved under that name.
  • CopyPreset() — Duplicates an existing preset with a new name. Useful when you want to make small tweaks to an existing setup without modifying the original.

Template Preset

A hidden template preset with id = -1 is automatically seeded during InitSchema() if it does not already exist. This template contains all 14 default channels with their events and filters pre-configured. It is never shown in the UI — GetPresetList() filters with id > 0.

When creating a new blank preset via CreateBlankPreset(), the plugin copies the template's channel structure rather than re-running all the seed INSERT statements. This is faster and ensures every new preset starts from a consistent baseline. If the template is somehow missing, it falls back to SeedDefaultChannels().

Per-Character Channel Overrides

When multiple characters share the same preset, their display preferences (tab order, enabled channels, pop-out settings) are stored separately in the char_channel_overrides table. The load and save flow works as follows:

  • LoadCharChannelOverrides() — After loading the preset's channel structure, overlays the character's saved display preferences onto each channel. Channels with no override row keep the preset's defaults.
  • SaveCharChannelOverrides() — Saves only the display preferences for the current character. The preset's structural data (events, filters) is not touched.

This separation means Character A can have Guild chat enabled with tab order 2, while Character B on the same preset has Guild chat disabled — without either affecting the other.

Switching Presets

Switching presets is a multi-step process:

  1. Save the current settings (so nothing is lost)
  2. SetActivePreset() — Updates the char_active_preset table to point to the new preset
  3. Unload — Tears down the current Blech events and clears in-memory channel data
  4. Load — Reads the new preset's channels, events, and filters from the database and registers fresh Blech events

This save-switch-unload-load sequence ensures a clean transition with no leftover state from the previous preset.

Loading and Saving

Loading Settings

LoadSettings() reads the full channel tree from the database using nested queries:

  1. Query all channels for the active preset
  2. For each channel, query its events
  3. For each event, query its filters

This builds up the complete in-memory representation of your chat configuration. After loading, the plugin registers Blech events for each enabled event pattern (see Chat Routing).

Saving Settings

SaveSettings() writes everything back to the database. It wraps the entire operation in a transaction for safety.

What is a transaction? A transaction groups multiple database operations into a single all-or-nothing unit. If any operation within the transaction fails, the entire batch is rolled back as if nothing happened. This prevents your database from ending up in a half-written state (for example, channels deleted but not yet re-inserted). 
BEGIN TRANSACTION;
-- Delete all existing channels for this preset (CASCADE removes events and filters too)
-- Re-insert all channels, events, and filters from memory
-- Update the preset's updated_at timestamp
COMMIT;

The "delete everything then re-insert" approach is simpler than trying to figure out what changed. Because the foreign keys use ON DELETE CASCADE, deleting the channels automatically removes their events and filters too. Since it all happens inside a transaction, either everything is saved or nothing is — you will not end up with a half-written state if something goes wrong.

After re-inserting all channels, the save also updates the preset's updated_at timestamp so you can see when a preset was last modified.

Global Settings

LoadGlobalSettings() and SaveGlobalSettings() handle the key-value pairs in the global_settings table. These are loaded separately from channel data because they apply regardless of which preset is active.

Saving uses an upsert strategy (INSERT OR REPLACE): if a key already exists for this character, its value is overwritten; if it does not exist yet, a new row is created. The entire batch of key-value writes is wrapped in a transaction for performance and safety.

Database Migration

MQMyChat started life as a Lua module (part of the MyUI project). If you previously used the Lua version, your settings are stored in a different location:

gPathConfig/MyUI/MyChat/MyChat.db

When MQMyChat initializes and finds that MyChat.db does not exist in the resources directory, it checks the old Lua module location. If the Lua database file is found, the plugin copies it to the new location. This is a one-time migration — after the copy, the plugin uses the new file exclusively.

Schema compatibility: The database schema is identical between the C++ plugin and the Lua module. No table structure changes are needed during migration — the file is copied as-is.

Schema Versioning

The database uses a version-gated migration system to evolve the schema safely. The schema_version table stores the current version number, and RunMigrations() applies any pending migrations in order.

How It Works

  1. InitSchema() creates all tables using CREATE TABLE IF NOT EXISTS
  2. RunMigrations() reads the current schema version
  3. Each migration is guarded by a version check: if (currentVersion < N)
  4. After each migration completes, the version is incremented

Migration History

Version Changes
0 → 1 Recreated char_channel_overrides table with per-character display columns (enabled, echo, main_enable, enable_links, pop_out, locked, scale, main_font_size, tab_order). Added lookup index.

Summary

Here is the big picture of how data flows through the database layer:

Plugin Loads │ ├─ Open() ──> Connect to SQLite, set pragmas ├─ InitSchema() ──> Create tables if missing ├─ RunMigrations() ──> Apply schema upgrades ├─ Seed template preset (id=-1) if missing ├─ Migration check ──> Copy Lua DB if needed ├─ GetOrCreatePreset() ──> Find or create active preset ├─ LoadSettings() ──> Channels → Events → Filters ├─ LoadGlobalSettings() ──> Key-value pairs └─ LoadCharChannelOverrides() ──> Per-char display prefs Plugin Runs │ ├─ SaveCharChannelOverrides() ──> Per-char display prefs ├─ SaveSettings() ──> Preset structure (events/filters) ├─ SaveGlobalSettings() ──> Upsert key-value pairs └─ Preset operations ──> Create, copy, rename, delete Plugin Unloads │ └─ Close() ──> WAL checkpoint, disconnect

For details on how the loaded data is used to route chat, see Chat Routing & Tokens. For how the settings UI lets you edit this data, see ImGui Rendering.
