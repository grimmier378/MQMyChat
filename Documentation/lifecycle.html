Plugin Lifecycle

This page walks through every stage of MQMyChat's life, from the moment MacroQuest loads the plugin DLL to the moment it unloads. If you are new to MQ plugin development, this is the right place to start. Every plugin follows the same callback pattern; MQMyChat simply fills in each callback with chat-specific logic.

Jargon you will see on this page:
  • TLO (Top-Level Object) — A named data accessor that macros and Lua scripts can query, like ${MyChat}. Think of it as a read-only API that other MQ code can call.
  • ConsoleWidget — An MQ ImGui widget that displays scrollable, colored text. Each chat tab in MQMyChat is backed by one of these.
  • Blech — MacroQuest's built-in text pattern matcher. It watches every chat line for patterns you define and fires a callback when one matches. See Chat Routing for full details.
  • Zep — The text editor library embedded inside each ConsoleWidget. You interact with it indirectly when you change font sizes.
  • pLocalPlayer — An MQ pointer to the currently logged-in character's spawn data. It is nullptr when no character is in the game world (for example, at the character select screen).
  • pLocalPC — Similar to pLocalPlayer, but points to the character's "player client" data (group info, inventory, etc.).
  • gPathResources — MQ's shared resources folder. By default, the resources subfolder within your MacroQuest directory.
  • gPathConfig — MQ's configuration folder. By default, the config subfolder within your MacroQuest directory.
  • GAMESTATE_INGAME / GAMESTATE_CHARSELECT — Integer constants that tell you whether EverQuest is on the character select screen or the player is actually in the game world.
Tip: All line numbers reference MQMyChat.cpp unless stated otherwise. Open it side-by-side while reading this page.

Plugin Callbacks (MQMyChat.cpp)

MacroQuest calls these functions automatically at the right time. You never call them yourself. Think of them as hooks: MQ says "something happened," and your plugin gets a chance to react.

InitializePlugin (line 25)

Called once when the plugin DLL is first loaded (typically at MQ startup or when you type /plugin MQMyChat). This is where all one-time setup happens: 

PLUGIN_API void InitializePlugin()
{
    g_chatEngine = new MyChatEngine();
    g_chatEngine->Initialize();
    AddCommand("/mychat", MyChatCommand);
    RegisterMyChatTLO();
}

Step by step:

  1. Creates the single MyChatEngine instance and stores it in the global g_chatEngine pointer.
  2. Calls Initialize(), which opens the SQLite database, checks for migration from the Lua version, creates the main console widget, the Blech instance, and an embedded Lua state (more on each below).
  3. Registers the /mychat slash command so players can type it in-game. See Commands & TLO for the full command table.
  4. Registers the TLO (${MyChat}) so macros and Lua scripts can query the plugin.

ShutdownPlugin (line 33)

Called once when the plugin is unloaded. Mirrors InitializePlugin in reverse order: unregister TLO, remove command, shut down the engine, delete it. 

PLUGIN_API void ShutdownPlugin()
{
    UnregisterMyChatTLO();
    RemoveCommand("/mychat");
    g_chatEngine->Shutdown();
    delete g_chatEngine;
    g_chatEngine = nullptr;
}

Shutdown() saves any unsaved settings, closes the Lua state, resets all console widgets, and closes the database. After this call the global pointer is nullptr, so every other callback checks for that before doing work.

SetGameState (line 42)

MacroQuest calls this whenever EverQuest transitions between game states (character select, in-game, server select, etc.). MQMyChat only cares about two: 

PLUGIN_API void SetGameState(int GameState)
{
    if (!g_chatEngine)
        return;

    if (GameState == GAMESTATE_INGAME)
        g_chatEngine->LoadCharacterSettings();
    else if (GameState == GAMESTATE_CHARSELECT)
        g_chatEngine->UnloadCharacterSettings();
}
  • GAMESTATE_INGAME — A character has entered the game world (finished loading into a zone). This is the signal to load that character's channels, filters, presets, and Blech events from the database.
  • GAMESTATE_CHARSELECT — The player returned to the character select screen. Save settings, tear down Blech events, and clear channel consoles so the next character starts fresh.

OnIncomingChat (line 53)

Fires for every line of chat that EverQuest itself generates (tells, say, group chat, combat messages, emotes, etc.). The plugin feeds the raw text into Blech for pattern matching. See Chat Routing & Tokens for what happens next. 

PLUGIN_API bool OnIncomingChat(const char* Line, DWORD Color)
{
    if (g_chatEngine)
        g_chatEngine->ProcessIncomingChat(Line, static_cast<int>(Color));
    return false;
}

Returning false means "I did not consume this line." Other plugins and MQ's own chat window still see it. MQMyChat never suppresses lines; it only copies them into its own consoles.

OnWriteChatColor (line 60)

Fires for every line written by MacroQuest or another plugin (via WriteChatf, WriteChatColor, etc.). Same idea as OnIncomingChat, but for MQ-originated output: 

PLUGIN_API void OnWriteChatColor(const char* Line, int Color, int Filter)
{
    if (g_chatEngine)
        g_chatEngine->ProcessWriteChat(Line, Color);
}

OnUpdateImGui (line 66)

Called every frame by the ImGui render loop. This is where all drawing happens. If the engine is not ready or the player is not in-game, the function bails out immediately. 

PLUGIN_API void OnUpdateImGui()
{
    if (!g_chatEngine || GetGameState() != GAMESTATE_INGAME)
        return;

    s_renderer.RenderMainWindow(*g_chatEngine);
    s_renderer.RenderPopOutWindows(*g_chatEngine);

    if (g_chatEngine->showConfigGUI)
        s_renderer.RenderConfigGUI(*g_chatEngine);

    if (g_chatEngine->showEditGUI)
        s_renderer.RenderEditChannelGUI(*g_chatEngine);
}

Notice that rendering is delegated to a separate MyChatRenderer object (s_renderer), not to methods on the engine itself. This keeps rendering logic out of the engine class. The renderer draws up to four things each frame:

  1. The main tabbed chat window (always drawn when in-game).
  2. Any pop-out channel windows (channels the user has detached into their own window).
  3. The config GUI, if the user toggled it on (via /mychat config or a button).
  4. The channel editor GUI, if the user is editing a specific channel.

For details on how each of these renders, see the ImGui Rendering page.

OnPulse (line 81)

Called roughly every frame. The plugin throttles its work to once every 500 ms using a static timer: 

PLUGIN_API void OnPulse()
{
    if (!g_chatEngine)
        return;

    static std::chrono::steady_clock::time_point pulseTimer
        = std::chrono::steady_clock::now();
    if (std::chrono::steady_clock::now() > pulseTimer)
    {
        pulseTimer = std::chrono::steady_clock::now()
            + std::chrono::milliseconds(500);
        g_chatEngine->CleanExpiredClaims();

        if (!g_chatEngine->charName.empty() &&
            g_chatEngine->m_lastSavedShowMain
                != g_chatEngine->showMainWindow)
        {
            g_chatEngine->SaveWindowVisibility();
        }
    }
}

Two things happen each tick:

  • CleanExpiredClaims() — Removes entries from m_claimedLines whose TTL (0.5 s) has elapsed. These entries track which lines were already routed to a channel, so the Spam channel knows what is unclaimed.
  • Autosave window visibility — If the user toggled the main window on or off (via the X button or /mychat hide), this detects the change and writes it to a per-character INI file immediately (see Window Visibility Persistence below).

MyChatEngine Lifecycle Methods

The plugin callbacks above delegate to these methods on MyChatEngine. Understanding them is key to understanding the plugin's startup and teardown sequence.

Initialize() (line 227)

One-time setup. Called from InitializePlugin. Here is the full method: 

void MyChatEngine::Initialize()
{
    database = std::make_unique<MyChatDatabase>();
    dbPath = fmt::format("{}\\MyChat.db", gPathResources);
    std::filesystem::path dbDir
        = std::filesystem::path(dbPath).parent_path();
    std::filesystem::create_directories(dbDir);

    if (!std::filesystem::exists(dbPath))
    {
        std::string luaDbPath
            = fmt::format("{}\\MyUI\\MyChat\\MyChat.db", gPathConfig);
        if (std::filesystem::exists(luaDbPath))
        {
            std::filesystem::copy_file(luaDbPath, dbPath);
            WriteChatf("\at[MQMyChat]\ax Migrated database from Lua MyChat: %s",
                luaDbPath.c_str());
        }
    }

    database->Open(dbPath);
    database->InitSchema();
    mainConsole = mq::imgui::ConsoleWidget::Create(
        "MQMyChat##MainConsole");
    m_blech = std::make_unique<Blech>('#', '|', BlechVarCallback);
    m_luaState = luaL_newstate();
    if (m_luaState)
        luaL_openlibs(m_luaState);
}

What each step does:

  1. Creates a MyChatDatabase helper object.
  2. Builds the database file path inside gPathResources (MQ's shared resources folder).
  3. Ensures the directory exists.
  4. Checks for migration from the Lua version (see below).
  5. Opens the SQLite database and initializes the schema (creates tables if missing). See Database & Settings for the schema.
  6. Creates the main console widget (the "All" tab).
  7. Creates a Blech instance. Blech is MQ's built-in pattern matcher. The '#' and '|' arguments configure the wildcard and separator characters.
  8. Creates an embedded LuaJIT state with standard libraries. This is used exclusively for Lua-style pattern matching in filters (see Pattern Matching).

Shutdown() (line 253)

void MyChatEngine::Shutdown()
{
    UnregisterBlechEvents();

    if (!charName.empty())
        SaveCharacterSettings();

    mainConsole.reset();

    for (auto& [id, ch] : settings.channels)
        ch.console.reset();

    if (m_luaState)
    {
        lua_close(m_luaState);
        m_luaState = nullptr;
    }

    database->Close();
}

Clean teardown in order: Blech events, save, consoles, Lua, database.

LoadCharacterSettings() (line 274)

Called when a character enters the game world. This is where per-character customization takes effect: 

void MyChatEngine::LoadCharacterSettings()
{
    if (!pLocalPlayer)
        return;

    charName = pLocalPlayer->Name;
    serverName = GetServerShortName();

    activePresetId = database->GetOrCreatePreset(serverName, charName);
    database->LoadSettings(activePresetId, settings);
    database->LoadGlobalSettings(serverName, charName, settings);
    LoadWindowVisibility();

    if (settings.channels.empty())
        PopulateDefaultChannels();

    for (auto& [id, ch] : settings.channels)
    {
        if (!ch.console)
            ch.console = mq::imgui::ConsoleWidget::Create(
                fmt::format("MQMyChat##Channel_{}", id).c_str());
    }

    RegisterBlechEvents();
    SortChannels();
    ApplyFontSizes();
    SaveCharacterSettings();
    database->GetPresetList(serverName, presetList);

    activePresetName.clear();
    for (const auto& p : presetList)
    {
        if (p.id == activePresetId)
        {
            activePresetName = p.name;
            break;
        }
    }
}

The sequence:

  1. Grab the character name and server short name from the game.
  2. Look up (or create) the active preset in the database.
  3. Load channel/event/filter settings from the preset.
  4. Load global settings (settings shared across presets).
  5. Load window visibility from the per-character INI file.
  6. If no channels exist yet, seed the defaults (PopulateDefaultChannels).
  7. Create a ConsoleWidget for every channel that doesn't have one.
  8. Register all enabled events with Blech.
  9. Sort channels by tab order for the UI.
  10. Apply saved font sizes to every console's Zep editor.
  11. Immediately save (to persist any defaults that were just created).
  12. Fetch the preset list and find the active preset name for display.

SaveCharacterSettings() (line 313)

void MyChatEngine::SaveCharacterSettings()
{
    if (activePresetId < 0)
        return;

    SyncFontSizes();
    database->SaveSettings(activePresetId, settings);
    database->SaveGlobalSettings(serverName, charName, settings);
    SaveWindowVisibility();
}

SyncFontSizes() reads the current pixel height from each Zep editor and writes it back into the settings struct, so what you see is what gets saved.

UnloadCharacterSettings() (line 324)

void MyChatEngine::UnloadCharacterSettings()
{
    if (charName.empty())
        return;

    SaveCharacterSettings();
    UnregisterBlechEvents();

    for (auto& [id, ch] : settings.channels)
        ch.console.reset();

    settings.channels.clear();
    charName.clear();
}

Saves, tears down Blech registrations, releases every channel console, empties the channel map, and clears the character name. The engine is now ready to load a different character.

DB Migration on First Load

MQMyChat is a C++ rewrite of the Lua-based MyChat module (part of the MyUI project). To make migration seamless, the plugin checks for the old database on first run:

  1. Initialize() builds the plugin's database path: gPathResources/MyChat.db.
  2. If that file does not exist, it looks for the Lua module's database at gPathConfig/MyUI/MyChat/MyChat.db.
  3. If found, it copies the file over with std::filesystem::copy_file.
  4. A chat message confirms the migration.
Why copy instead of move? The Lua module may still be installed. Copying preserves the original so users can go back if needed.

Because both versions share the same SQLite schema, channels, events, filters, and presets carry over with no conversion step.

Window Visibility Persistence

Whether the main window is shown or hidden is stored separately from the SQLite database. This is a deliberate design choice: the database schema must stay compatible with the Lua module, and window visibility is a C++ plugin concept that does not exist in the Lua version.

Where it is stored

A per-character INI file at: 

gPathConfig/MQMyChat/<Server>/<CharName>.ini

Inside that file, a single section and key: 

[Windows]
ShowMain=1

Load and save

void MyChatEngine::LoadWindowVisibility()
{
    if (!pLocalPC)
        return;

    std::string path
        = GetWindowVisibilityIniPath(serverName, charName);
    if (path.empty())
        return;

    showMainWindow
        = GetPrivateProfileBool("Windows", "ShowMain", true, path);
    m_lastSavedShowMain = showMainWindow;
}

void MyChatEngine::SaveWindowVisibility()
{
    std::string path
        = GetWindowVisibilityIniPath(serverName, charName);
    if (path.empty())
        return;

    WritePrivateProfileBool("Windows", "ShowMain",
        showMainWindow, path);
    m_lastSavedShowMain = showMainWindow;
}

Autosave in OnPulse

The OnPulse callback compares m_lastSavedShowMain to the current value of showMainWindow every 500 ms. If they differ, it calls SaveWindowVisibility(). This way the user never loses their show/hide preference, even if the plugin crashes or MQ is closed abruptly.

Blech Event Registration

As mentioned in the glossary above, Blech is MacroQuest's built-in text pattern matcher. It uses a simple wildcard syntax (not regex) to watch every chat line for patterns you define. When a line matches, Blech fires a callback so the plugin knows which channel to route the line to. For full syntax details, see Chat Routing & Tokens.

MQMyChat registers one Blech event per enabled event string in each enabled channel.

RegisterBlechEvents() (line 372)

void MyChatEngine::RegisterBlechEvents()
{
    if (!m_blech)
        return;

    m_blech->Reset();
    m_blechEventMap.clear();

    for (auto& [channelId, ch] : settings.channels)
    {
        if (!ch.enabled)
            continue;

        for (auto& evt : ch.events)
        {
            if (!evt.enabled || evt.eventString.empty())
                continue;

            auto* data = new BlechEventData{
                channelId, evt.eventIndex };
            unsigned int id = m_blech->AddEvent(
                evt.eventString.c_str(), BlechCallback, data);
            evt.blechId = id;
            m_blechEventMap[id] = *data;
            delete data;
        }
    }
}

Key points:

  • Always starts by resetting Blech (removes all previous events). This makes it safe to call repeatedly — for example, after the user edits channels in the config GUI.
  • Skips disabled channels and disabled events.
  • Each registered event stores a BlechEventData containing the channelId and eventIndex. When Blech matches a line, the callback uses these to know which channel to route to.
  • The returned Blech ID is saved in m_blechEventMap for reverse lookup and in evt.blechId for reference.

UnregisterBlechEvents() (line 399)

void MyChatEngine::UnregisterBlechEvents()
{
    if (!m_blech)
        return;

    m_blech->Reset();
    m_blechEventMap.clear();
}

Simple: reset Blech and clear the map. Called during UnloadCharacterSettings() and Shutdown().

When registration happens

Blech events are registered (or re-registered) at these points:

  • LoadCharacterSettings() — Character enters the game.
  • After saving from the Edit GUI — The user may have changed event strings.
  • /mychat reload — Calls UnloadCharacterSettings then LoadCharacterSettings.
