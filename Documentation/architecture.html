Architecture Overview

Last Updated: 2026-04-24

This page explains how MQMyChat is structured: what the major pieces are, how data flows through them, and how the source files relate to each other. If you are new to the codebase, start here before diving into specific topics.

High-Level Design

At the center of everything is the MyChatEngine class. Think of it as the brain of the plugin. It owns and coordinates all the major subsystems:

  • Blech pattern matcher — A fast text-matching engine bundled with MacroQuest. It watches every line of incoming chat and checks it against your configured patterns. Patterns use #*# as a wildcard that matches any amount of text, and #1#, #2#, etc. as numbered capture wildcards that also pass the matched text to a callback. (See the Blech pattern reference for full syntax.)
  • SQLite database (via MyChatDatabase) — A small self-contained database stored as a single file on disk. It holds all your settings, channels, events, and filters so they persist between sessions.
  • Embedded LuaJIT state — A Lua interpreter (specifically LuaJIT, a fast Lua 5.1 implementation) for running custom scripts in response to chat events.
  • ConsoleWidget instances — One per channel. A ConsoleWidget is MacroQuest's wrapper around the Zep text editor library that provides a scrollable, colored text display area (like a terminal window). This is where chat lines actually appear.
  • MyChatRenderer — Handles all ImGui drawing. ImGui ("Immediate Mode GUI") is the graphical interface library MacroQuest uses — it redraws windows, tabs, buttons, and text every frame. The renderer draws the chat windows, the settings GUI, and the channel editor.

The engine does not render anything directly. It processes chat, manages state, and hands data to the renderer. All text output — from Blech-routed lines, the spam channel, the SendToChannel() API, and local echo in the renderer — passes through a single method: AppendToConsole(). This method handles timestamp injection (stripping duplicate timestamps and optionally prepending colored ones) before writing to the underlying ConsoleWidget. The renderer header only forward-declares MyChatEngine — it does not include channel data structures directly. The renderer's .cpp file includes MQMyChat.h and MyChatDatabase.h to access the full types, but the separation at the header level keeps dependencies clean. The renderer does not know about Blech — it just reads the channel data and draws it.

Data Flow

Here is how a single line of chat travels from EverQuest to your screen through MQMyChat. Follow the arrows from top to bottom:

EverQuest Client | v +------------------------------+ | OnIncomingChat / | MQ hooks these callbacks and | OnWriteChatColor | passes every chat line to the plugin. +------------------------------+ | v +------------------------------+ | Blech::Feed() | The raw chat text is fed into the | | Blech pattern matcher. +------------------------------+ | | (pattern matches) v +------------------------------+ | BlechCallback() | When a pattern matches, this callback | | fires with the matched event ID. +------------------------------+ | v +------------------------------+ | RouteToChannel() | Looks up which channel(s) the matched | | event belongs to. +------------------------------+ | v +------------------------------+ | Filter Matching | Each event can have filters. The chat | | line is checked against filter strings | | to determine color and inclusion. +------------------------------+ | v +------------------------------+ | AppendToConsole() | Central output method. Strips any | | existing [HH:MM:SS] timestamps, | | optionally prepends a colored | | timestamp via multiple AppendText() | | calls with explicit MQColor values, | | preserving the filter-assigned color. +------------------------------+ | v +------------------------------+ | MyChatRenderer draws the | ImGui renders the console widget | channel tab in the window | inside a tab within a chat window. +------------------------------+

The important thing to understand is that chat flows in one direction. EQ produces it, Blech matches it, the engine routes it, and the renderer displays it. There is no back-and-forth.

File-by-File Breakdown

Now that you understand the big picture, here is how the source files map to it. For each file, we explain what it does and how it connects to the others.

MQMyChat.h + MQMyChat.cpp — The Engine

These two files define MyChatEngine, the central class. The header declares the class with its member variables (the Blech instance, the database, the Lua state, the settings struct) and its public methods. The cpp file implements everything:

  • Plugin lifecycle hooks: InitializePlugin, ShutdownPlugin, SetGameState, OnPulse. These are called by MQ at specific moments (see Plugin Lifecycle).
  • Chat hooks: OnIncomingChat and OnWriteChatColor. These intercept chat from EQ.
  • Blech setup: Registering patterns with the Blech matcher and defining the callback that fires on matches. Includes RefreshBlechEvents(), a convenience method that unregisters and re-registers all Blech events in one call — used by the UI when toggling channels or events on/off so changes take effect immediately without requiring a full save or reload.
  • Console output: The AppendToConsole() method, which is the single chokepoint for all text written to any ConsoleWidget. It handles timestamp injection (stripping duplicate timestamps and optionally prepending colored ones via multiple AppendText calls with explicit MQColor values per segment, preserving the filter-assigned message color) before forwarding to the widget. See AppendToConsole and Timestamps.
  • Token substitution: Replacing tokens like M3 in filter patterns with live game values before matching.
  • Command handling: Processing /mychat slash commands.

This file includes MyChatChannel.h (for data structures), MyChatDatabase.h (for persistence), MyChatRenderer.h (for rendering), and MyChatTLO.h (for TLO registration).

MyChatChannel.h — Data Structures

This is a header-only file with no corresponding cpp. It defines the core data types that everything else uses:

  • ChatFilter — A string to match against chat text, plus a color to apply when it matches. Each filter can also be individually enabled or hidden.
  • ChatEvent — A Blech pattern string plus a list of ChatFilters. When the pattern matches, the filters determine what color to use. Also tracks its own Blech registration ID.
  • ChatChannel — A named group of ChatEvents. Each channel has its own ConsoleWidget for display. Think of a channel as one tab in a chat window. Channels also carry per-channel settings like echo command, font size, scale, pop-out window support, locking, link detection, and a command input history buffer.
  • PresetInfo — Metadata for a saved preset: its ID, name, server name, and who created it. Presets let you save and load entire channel configurations.
  • MyChatSettings — The top-level settings container. Holds a map of all channels, global options (theme, timestamps, font size, auto-scroll, local echo, hotkey binding), and window configurations.

This file also defines important constants (see the Constants section below).

MyChatDatabase.h + MyChatDatabase.cpp — Persistence

The database layer. MyChatDatabase wraps SQLite3 and provides methods to load and save the entire settings tree. The cpp file is one of the largest in the project (~960 lines) because it handles:

  • Creating the database schema (tables for channels, events, filters, settings)
  • Seeding default presets so new users get a reasonable starting configuration
  • Loading everything on startup and saving changes back
  • Schema migration when the database format changes between versions
  • Per-character channel display overrides (separate from preset structure)
  • Template preset seeding for consistent blank preset creation

For a deep dive, see Database & Settings.

MyChatRenderer.h + MyChatRenderer.cpp — The UI

MyChatRenderer handles all ImGui drawing. The header file only forward-declares MyChatEngine — it does not directly include any data structure headers. The .cpp file includes MQMyChat.h and MyChatDatabase.h to access the full types. It reads from the MyChatSettings structure (it does not own the data — the engine does) and draws:

  • Chat windows with tab bars (one tab per channel)
  • The settings/configuration panel
  • The channel editor (add/remove/edit channels, events, filters)
  • Tab reordering via drag-and-drop
  • Theme selection
  • Preset management (copy, create blank, rename, delete)

At ~1115 lines, this is the second-largest file. For details, see ImGui Rendering.

MyChatTLO.h + MyChatTLO.cpp — Macro Interface

These files expose plugin data to MQ macros through the Top Level Object (TLO) system. A TLO is MacroQuest's way of letting macro scripts read data from plugins — think of it as a read-only API that macros can call using expressions like ${MyChat.Channel[Tells].Name}. This is how macro scripts and Lua scripts can interact with MQMyChat:

  • MyChatType — The root TLO. Lets macros query global state.
  • MyChatChannelType — Represents a single channel. Lets macros read channel properties.
  • Send method: Allows macros to inject text directly into a channel, bypassing the normal Blech routing.

See Commands & TLO for usage examples.

Theme.h — Visual Themes

A large header file (868 lines) containing 11 ImGui color themes. Each theme is a function that sets ImGui style colors. The renderer calls into this file when the user selects a different theme from the settings panel.

CMakeLists.txt — Build Configuration

A short (34-line) CMake file that defines the build target, links against MQ2Main, and finds the required packages (SQLite3, Lua). Nothing unusual here — it follows the standard MQ plugin CMake pattern.

Key Classes

Here is a summary of the main classes and what they are responsible for:

Class Defined In Responsibility
MyChatEngine MQMyChat.h/.cpp Central coordinator. Owns the Blech matcher, database, Lua state, and settings. Handles chat routing, commands, and plugin lifecycle.
MyChatDatabase MyChatDatabase.h/.cpp SQLite wrapper. Loads and saves all settings. Seeds defaults for new installs.
MyChatRenderer MyChatRenderer.h/.cpp ImGui rendering. Draws chat windows, tabs, settings GUI, and the channel editor.
MyChatType MyChatTLO.h/.cpp Root TLO type. Exposes plugin data to macros.
MyChatChannelType MyChatTLO.h/.cpp Channel TLO type. Exposes individual channel data to macros.

Key Data Structures

These are all defined in MyChatChannel.h and are used throughout the codebase. Understanding them is essential to understanding the plugin:

ChatFilter

The smallest unit of matching. A filter has a match string, a color, an enabled flag, and a hidden flag. When a chat line contains the match string, the line is displayed in that color. If a chat event has multiple filters, the first one that matches determines the color. Disabled or hidden filters are skipped during matching.

ChatEvent

Represents a Blech pattern plus its associated filters. The pattern is the broad match (like "#*# tells you, '#*#'") that Blech checks against every incoming line. Remember that #*# is not a single-character wildcard — it matches any amount of text (zero or more characters). When the pattern fires, the event's filters provide fine-grained color control. Each event also stores its Blech registration ID (blechId) so it can be unregistered later.

ChatChannel

A named collection of events. Each channel has:

  • A name (displayed as a tab label)
  • A list of ChatEvents that feed into it
  • A ConsoleWidget instance that displays the matched chat
  • An echo command (defaults to /say) — the slash command used when you type into the channel's input box
  • Display settings: font size, scale factor, link detection, pop-out window mode, and a lock flag to prevent accidental changes
  • A command history buffer (up to MAX_COMMAND_HISTORY entries) and an input text buffer for the chat input box
  • A tab order value that controls where the channel appears among tabs

PresetInfo

Metadata about a saved configuration preset. Each preset has an ID, a human-readable name, the server it was created on, and the character who created it. Presets let users save their entire channel setup and reload it later, or share it across characters on the same server. See Database & Settings for how presets are stored and loaded.

MyChatSettings

The top-level container for all plugin state. It holds:

  • A std::map of channel ID to ChatChannel
  • Global settings: theme index, timestamp toggle (timeStamps, defaults to true — controls whether AppendToConsole() prepends colored timestamps via multiple AppendText calls with explicit MQColor values per segment to every line), global scale, default echo command, default font size, link detection, auto-scroll, local echo, and a configurable hotkey for focusing the chat input
  • A window lock flag that prevents all windows from being moved or resized

Important Constants

These constants, defined in MyChatChannel.h, control various behaviors:

Constant Value Purpose
CHANNEL_SPAM 9000 The ID of the built-in spam channel. All unmatched chat goes here. This is a catch-all so nothing is lost.
CHANNEL_RESERVED_START 9000 Channel IDs at or above this value are reserved for system use. User-created channels should have IDs below this.
CLAIMED_TTL 0.5f Time-to-live in seconds for a "claimed" chat line. When a line is matched by an event, it is marked as claimed for this duration to prevent duplicate routing.
MAX_COMMAND_HISTORY 100 Maximum number of commands stored in the input history. The chat input box keeps a rolling history so you can press up/down to recall previous commands.
MAX_CHANNEL_HISTORY 10 Maximum number of recent channel entries kept per channel. Limits how many history entries each channel retains.
TAB_ORDER_SAVE_INTERVAL 3.0f How often (in seconds) the plugin saves tab order changes to the database. Tab reordering happens frequently via drag-and-drop, so saving is debounced to avoid excessive disk writes.

Dependency Diagram

This shows which files include which. Arrows mean "includes" or "depends on":

CMakeLists.txt | | (builds) v MQMyChat.cpp ------> MQMyChat.h --------> MyChatChannel.h | | ^ | +--------> Blech.h | | | +----> MyChatDatabase.h ------------------->+ | | | | v | | MyChatDatabase.cpp | | | +----> MyChatRenderer.h | | | (data structures used | v by all other files) | MyChatRenderer.cpp ----> MQMyChat.h --+ | | | | +----> MyChatDatabase.h | | v | | Theme.h | | | +----> MyChatTLO.h | | | v | MyChatTLO.cpp | Note: MyChatRenderer.h only forward-declares MyChatEngine (no direct include of MyChatChannel.h). The .cpp file gets the full types through MQMyChat.h.

The key thing to notice is that MyChatChannel.h is at the bottom of the dependency tree. Nearly every other file depends on it because it defines the data structures that flow through the entire system. If you change a data structure there, you will likely need to update code in multiple files.

MQMyChat.cpp sits at the top — it pulls everything together. The database, renderer, and TLO modules are relatively independent of each other; they communicate through the shared MyChatSettings data and the engine that owns it.
